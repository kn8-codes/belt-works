# belt.works Site Direction

Source of truth for the next rebuild pass. Read `CONTEXT.md` first. This file is the working synthesis: the context file sets constraints; this document sets the path.

## Positioning

belt.works is a software research lab.

The research produces software. The software is the evidence.

Some of it becomes tools. Some becomes services. Some becomes client work. All of it tests the same hypothesis:

> Serious technical work can happen outside institutions, with consumer hardware, open tools, and working-class constraints. The work either holds up or it does not.

## Short Form

Research first. Software as evidence.

Consumer hardware. Consumer spend. Business results.

## What the Website Must Do

1. Explain what belt.works is.
2. Show proof through shipped or near-shipped work.
3. Funnel people who want help toward FOUNDRY intake.
4. Funnel people who want tools toward Intel, IOC, Data, Rooms, BoomMates, etc.
5. Collect a mailing list signup as the primary low-friction CTA.

## What the Website Must Not Do

- Do not sound like a startup.
- Do not sound like an agency.
- Do not sell Akron as tragedy or triumph.
- Do not motivate, inspire, or perform grit.
- Do not over-explain the philosophy before showing useful work.
- Do not pretend every project is fully finished.
- Do not write vague AI copy.

## Audience

Anybody with a real problem who wants an affordable software path.

That includes:

- Small operators who cannot afford enterprise software.
- People with messy operational problems.
- Local businesses needing useful automation, ecommerce, forms, dashboards, or data.
- Technical/prosumer users who need OSINT, IOC, scraping, normalization, or APIs.
- Collaborators who understand the method and want to work near it.

## Voice Rules

Useful. Utilitarian. Determined.

Write like the tool already exists and has to survive use.

Good:

- "Bring a problem."
- "We are doing the research."
- "The research produces software."
- "The work either holds up or it does not."
- "Consumer hardware. Consumer spend. Business results."

Bad:

- "Transform your business with AI."
- "Empowering innovation."
- "Akron's scrappy underdog tech story."
- "Disrupting the future."
- "Scalable next-generation solutions."

## Homepage Spine

1. **Hero**
   - `belt.works`
   - `Research first. Software as evidence.`
   - Subline: `Consumer hardware. Consumer spend. Business results.`
   - Primary CTA: mailing list
   - Secondary CTA: bring a problem / FOUNDRY intake

2. **What this is**
   - Software lab.
   - The output is evidence.
   - Not a startup, not an agency, not a motivational story.

3. **Proof**
   - ShopFloor
   - BoomMates
   - ROOMS
   - GREYMARKET
   - hosp-or-not / Hospnaught
   - OBDsidian / mesh work where relevant

4. **Services / properties**
   - data.belt.works — scraping, normalization, API
   - intel.belt.works — OSINT/threat intel
   - ioc.belt.works — indicators, actor profiles, stories
   - rooms.belt.works — public sanitized agent/mesh view
   - FOUNDRY — client intake methodology

5. **The method**
   - One person, three machines, open tools.
   - Infrastructure enables autonomous sessions.
   - Autonomous sessions build products.
   - Products generate evidence and revenue.

6. **Mailing list**
   - Primary CTA.
   - One send per month.
   - Plain update: what shipped, what broke, what is next.

7. **Bring a problem**
   - Secondary CTA.
   - Routes to FOUNDRY intake.

- `/method` — mesh, agents, infrastructure, process
- `/rooms` — public sanitized ROOMS view or placeholder spec
- `/services` — affordable problem-solving paths, not agency packages
- `/data` — scraping/normalization/API service
- `/intel` — OSINT/threat intelligence direction
- `/ioc` — IOC/threat actor data direction
- `/foundry/intake` — existing intake flow

## Status Labels

Use honest status language:

- Live
- Shippable
- Built enough to show
- Prototype
- On hold
- Needs wiring
- Waiting on client/spec/hardware

Do not use "coming soon" unless there is no better truth.

## Design Direction

Dark. Technical. Minimal. Hardware visible where possible.

The site should feel like a live lab bench, not a pitch deck.

Use:

- monospace details
- status panels
- project cards
- plain labels
- logs/telemetry where sanitized
- GitHub links

Avoid:

- glossy SaaS gradients
- fake dashboard screenshots
- over-animated hero sections
- generic AI icons
